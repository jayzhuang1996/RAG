The Two Modes: Ingest and Query

The app has two completely separate jobs — like a library system that has both a cataloguing department and a reference desk. They never run at the same time.

📥

Ingest Mode

Run once (or weekly). Scrapes YouTube channels, downloads transcripts, chops text into searchable pieces, and stores them in the database. This is the cataloguing department — slow, thorough, runs in the background.

python main.py ingest --limit 50
❓

Query Mode

Run anytime. Takes your question, searches the already-processed database, and returns an answer in seconds. This is the reference desk — fast, because all the hard work was done upfront.

python main.py query "your question"
🏗️
Why separate these?

Processing 200 hours of video takes 30 minutes and costs money (transcription API calls). Answering a question takes 3 seconds. Keeping them separate means you pay the big cost once, then query for free as many times as you want.

The Two Databases

Your app actually uses two different storage systems, and each is optimized for a different kind of search. Think of them as two different card catalogues in the same library:

🗄️

ChromaDB (or Supabase)

Stores text chunks as vectors — numbers that capture the meaning of each passage. Finds results that are semantically similar to your question, even if they use different words.

📝

SQLite / BM25 Index

Stores raw text and builds a keyword index. BM25 finds results that contain the exact words from your question — great for proper nouns like names and company names.

💡
Why use both?

If you search "Altman on AGI timeline," vector search finds semantically related passages even if they say "artificial general intelligence schedule." BM25 finds passages that literally contain "Altman." Using both together — called hybrid search — catches results that either method alone would miss.

The Clever Trick: Parent-Child Chunks

Here is the most interesting engineering decision in the whole codebase. When you slice a transcript, you don't just create one set of chunks — you create two:

👶

Child Chunks (Small)

~250 tokens (~1,000 characters). These are what get embedded into vectors and searched. Small = better semantic precision, fits inside the embedding model's limit.

👨

Parent Chunks (Large)

~750 tokens (~3,000 characters). These are what get returned to the AI as context. Large = more surrounding context, so the AI has enough material to give a complete answer.

💡
The "search small, return big" principle

You search with the small child chunk (precise match), but you return the large parent chunk (full context). It's like finding a specific page in a book using the index, then reading the whole chapter around it — not just the one sentence.

What is an Embedding?

After chunking, every piece of text needs to be converted into a "vector" — a list of numbers — before it can be stored in ChromaDB. This is the job of embeddings.py using a model called MiniLM.

🔢
Vectors in plain English

Imagine placing every sentence in a 384-dimensional space (like a map, but with 384 axes instead of 2). Two sentences that mean the same thing end up close together on this map. When you ask a question, the system converts your question into a location, then finds the transcript chunks sitting closest to it. That's semantic search.

Module 4

Finding the Right Answer

Why your app uses three different search methods at once — and how they work together.

The Problem with a Single Search

Imagine you're trying to find a quote by Sam Altman about "making AI systems that don't hurt people." You have two tools:

  • Keyword search — great if it says "Sam Altman" literally, terrible if he's referred to as "the OpenAI CEO"
  • Semantic/vector search — great for understanding meaning ("not hurt" ≈ "safety"), terrible for exact names and rare terms

Neither alone is reliable. Your app uses both together — a technique called hybrid search followed by a neural reranker. Here is the three-step process:

Step 1 + 2: Parallel Search and Fusion

The HybridRetriever.retrieve() method runs BM25 and vector search in parallel, then fuses their ranked lists using an algorithm called RRF:

CODE 

def retrieve(self, query):
  bm25 = self.search_bm25(query)
  vector = self.search_vector(query)
  fused = self.rrf_fusion(bm25, vector)
  reranked = self.rerank(query, fused)
  return self.get_context(reranked)
PLAIN ENGLISH

The main retrieval function — called every time you ask a question.

Run a keyword search — find chunks that contain the exact words you used.

Run a semantic search — find chunks that are meaningfully similar to your question.

Merge both ranked lists — chunks appearing near the top of both get promoted.

Pass the merged list to Cohere AI for a final neural reranking pass.

Return the parent chunks (full context) for each top result.

Step 3: The Reranker — Your Quality Filter

After BM25 + vector search gives you 20 candidates, the fused list is passed to Cohere Rerank. This is the most important quality step.

🏆
Why reranking matters

BM25 and vector search are fast but imprecise — they're doing math, not reading. Cohere actually reads your question and each candidate passage together before deciding which is most relevant. It catches the difference between a passage that mentions "AI safety" in passing vs. one where it's the entire subject.

The constants that control this are set at the top of retrieval.py:

VECTOR_TOP_K = 20 Vector search returns its top 20 candidates before fusion
BM25_TOP_K = 20 Keyword search also returns 20 candidates
RERANK_TOP_N = 5 Cohere reranks all candidates and returns the final top 5 for the AI
🎛️
Tuning tip for you

If your answers feel incomplete, try increasing RERANK_TOP_N to 7-8. This passes more context to the AI. If answers feel unfocused or too long, decrease it. This is a dial you control.

Function 1: chunk_text() — Lines 1–42

This is the low-level slicer. It takes a long string and cuts it into overlapping pieces of a fixed character size. Every other function in the codebase calls this one.

CODE · chunking.py 

import uuid

def chunk_text(text, chunk_size, overlap):
  """
  Simple character-based chunker.
  Args:
      text (str): Input text.
      chunk_size (int): Size in characters.
      overlap (int): Overlap in characters.
  Returns:
      list of dicts: [{'text': str, ...}]
  """
  if not text:
    return []

  chunks = []
  start = 0
  text_len = len(text)

  while start < text_len:
    end = min(start + chunk_size, text_len)
    chunk_text = text[start:end]

    if end < text_len:
      last_space = chunk_text.rfind(' ')
      if last_space != -1 and last_space > chunk_size * 0.5:
        end = start + last_space + 1
        chunk_text = text[start:end]

    chunks.append({
      'text': chunk_text,
      'start_char': start,
      'end_char': end
    })

    start += (chunk_size - overlap)

  return chunks

Function 2: create_parent_child_chunks() — Lines 44–103

The orchestrator. Calls chunk_text() twice per transcript — once to make large parent chunks, then again on each parent to make small child chunks. Every child knows its parent's ID.

CODE · chunking.py 

def create_parent_child_chunks(video_id, full_text):
  """
  Generates Parent and Child chunks for a video.
  Strategy:
    1. Create Parent Chunks (Large context).
    2. Chunk each Parent into Children.
    Sizes:
    - Parent: ~3000 chars (~750 tokens)
    - Child: ~1000 chars (~250 tokens)
    - Overlap: 200 chars
  """

  parent_chunks_data = chunk_text(
    full_text,
    chunk_size=3000,
    overlap=200
  )

  all_chunks = []

  for p_idx, p_data in enumerate(parent_chunks_data):
    parent_id = str(uuid.uuid4())

    all_chunks.append({
      'id': parent_id,
      'video_id': video_id,
      'text': p_data['text'],
      'type': 'parent',
      'parent_id': None,
      'chunk_index': p_idx,
      'start_char': p_data['start_char'],
      'end_char': p_data['end_char']
    })

    child_chunks_data = chunk_text(
      p_data['text'],
      chunk_size=1000,
      overlap=100
    )

    for c_idx, c_data in enumerate(child_chunks_data):
      child_id = str(uuid.uuid4())
      all_chunks.append({
        'id': child_id,
        'video_id': video_id,
        'text': c_data['text'],
        'type': 'child',
        'parent_id': parent_id,
        'chunk_index': c_idx,
        'start_char': p_data['start_char'] + c_data['start_char'],
        'end_char': p_data['start_char'] + c_data['end_char']
      })

  return all_chunks

build_bm25_index() & search_bm25() — Lines 31–61

  def build_bm25_index(self):
    """Load child chunks from DB and build index."""
    if USE_SUPABASE:
      res = self._supabase.table("viking_chunks")
        .select("id, text").eq("type", "child").execute()
      rows = res.data
    else:
      conn = get_connection()
      cursor = conn.cursor()
      cursor.execute("SELECT id, text FROM chunks WHERE type = 'child'")
      rows = cursor.fetchall()
      conn.close()

    self._bm25_corpus = [
      {'id': r['id'], 'tokens': r['text'].lower().split()}
      for r in rows
    ]
    tokenized = [c['tokens'] for c in self._bm25_corpus]
    self._bm25_index = BM25Okapi(tokenized)
    print(f"BM25 index built: {len(self._bm25_corpus)} child chunks.")

  def search_bm25(self, query, top_k=BM25_TOP_K):
    if not self._bm25_index:
      self.build_bm25_index()
    tokens = query.lower().split()
    scores = self._bm25_index.get_scores(tokens)
    scored = sorted(
      zip([c['id'] for c in self._bm25_corpus], scores),
      key=lambda x: x[1],
      reverse=True
    )
    return [
      {'id': chunk_id, 'score': float(score)}
      for chunk_id, score in scored[:top_k]
    ]

  def search_vector(self, query, top_k=VECTOR_TOP_K):
    model = get_embedding_model()
    query_embedding = model.encode([query]).tolist()[0]

    if USE_SUPABASE:
      res = self._supabase.rpc('match_chunks', {
        'query_embedding': query_embedding,
        'match_threshold': 0.1,
        'match_count': top_k
      }).execute()
      return [{'id': r['id'], 'score': r['similarity']} for r in res.data]
    else:
      collection = get_collection()
      results = collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k,
        include=["distances"]
      )
      ids = results['ids'][0]
      scores = [1 / (1 + d) for d in results['distances'][0]]
      return [{'id': cid, 'score': s} for cid, s in zip(ids, scores)]

  def rrf_fusion(self, bm25_results, vector_results, k=60):
    scores = {}
    for rank, item in enumerate(bm25_results):
      scores[item['id']] = scores.get(item['id'], 0.0) + 1.0 / (k + rank + 1)
    for rank, item in enumerate(vector_results):
      scores[item['id']] = scores.get(item['id'], 0.0) + 1.0 / (k + rank + 1)
    fused = sorted(scores.items(), key=lambda x: x[1], reverse=True)
    return [{'id': cid, 'score': score} for cid, score in fused]

import yaml

def get_config_prompt():
  config_path = os.path.join(
    os.path.dirname(os.path.dirname(
      os.path.abspath(__file__))),
    'config', 'prompts.yaml'
  )
  with open(config_path, 'r') as f:
    config = yaml.safe_load(f)
  return config.get('query_system_prompt', "Answer based on context.")

def generate_answer(query: str) -> str:
  """Full RAG pipeline: retrieve → prompt → call Moonshot → return."""
  system_prompt = get_config_prompt()
  retriever = get_retriever()

  print("Retrieving relevant context...")
  context_results = retriever.retrieve(query)

  if not context_results:
    return "No relevant content found in the knowledge base."

  context_text, sources = _build_context_block(context_results)

  user_message = f"""Context from podcast transcripts:
{context_text}
---
Question: {query}
Answer (cite sources as [Source N]):
"""

  moonshot_key = os.getenv("MOONSHOT_API_KEY")
  if not moonshot_key:
    return "Backend Error: MOONSHOT_API_KEY missing."

  client = OpenAI(
    api_key=moonshot_key,
    base_url="https://api.moonshot.ai/v1"
  )

  response = client.chat.completions.create(
    model="moonshot-v1-128k",
    messages=[
      {"role": "system", "content": system_prompt},
      {"role": "user",   "content": user_message}
    ],
    temperature=0.3,
    stream=True
  )

  print(f"\n{'='*60}\nQuery: {query}\n{'='*60}")
  print("\nSources used:")
  for s in sources:
    print(f"  [{s['index']}] {s['title']}")
  print(f"\n{'='*60}\nAnswer:\n")

  full_answer = ""
  for chunk in response:
    delta = chunk.choices[0].delta.content or ""
    print(delta, end="", flush=True)
    full_answer += delta

  print(f"\n{'='*60}\n")
  return full_answer
