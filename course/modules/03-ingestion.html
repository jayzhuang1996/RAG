The Ingestion Pipeline

The Clever Trick: Parent-Child Chunks

Here is the most interesting engineering decision in the whole codebase. When you slice a transcript, you don't just create one set of chunks — you create two:

👶

Child Chunks (Small)

~250 tokens (~1,000 characters). These are what get embedded into vectors and searched. Small = better semantic precision, fits inside the embedding model's limit.

👨

Parent Chunks (Large)

~750 tokens (~3,000 characters). These are what get returned to the AI as context. Large = more surrounding context, so the AI has enough material to give a complete answer.

💡
The "search small, return big" principle

You search with the small child chunk (precise match), but you return the large parent chunk (full context). It's like finding a specific page in a book using the index, then reading the whole chapter around it — not just the one sentence.

The Code Behind Chunking

Here is the exact function from chunking.py that creates parent and child chunks:

CODE 

def create_parent_child_chunks(video_id, full_text):
  parent_chunks = chunk_text(
    full_text,
    chunk_size=3000,
    overlap=200
  )
  for p_data in parent_chunks:
    parent_id = str(uuid.uuid4())
    child_chunks = chunk_text(
      p_data['text'],
      chunk_size=1000,
      overlap=100
    )
PLAIN ENGLISH

Define a function that takes a full transcript and slices it up.

First pass: create big "parent" slices — each about 3,000 characters (750 words).

The 200-character overlap means consecutive chunks share some text at their edges — so a sentence split between two chunks isn't lost.

For each parent slice, now go through it again...

Give this parent a unique ID (like a random serial number).

Second pass: chop each parent into smaller "child" slices — 1,000 characters each.

Each child inherits its parent's ID, creating the family tree relationship.

What is an Embedding?

After chunking, every piece of text needs to be converted into a "vector" — a list of numbers — before it can be stored in ChromaDB. This is the job of embeddings.py using a model called MiniLM.

🔢
Vectors in plain English

Imagine placing every sentence in a 384-dimensional space (like a map, but with 384 axes instead of 2). Two sentences that mean the same thing end up close together on this map. When you ask a question, the system converts your question into a location, then finds the transcript chunks sitting closest to it. That's semantic search.
