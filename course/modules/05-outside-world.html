The Dual-Database Architecture

One of the most interesting architectural decisions: your app supports two different database modes, switchable via a single environment variable. This is called the Strategy pattern.

CODE — retrieval.py 

USE_SUPABASE = os.getenv(
  "USE_SUPABASE", "false"
).lower() == "true"

if USE_SUPABASE:
  res = self._supabase
    .table("viking_chunks")
    .select("id, text")
    .execute()
else:
  cursor.execute(
    "SELECT id, text FROM chunks"
  )
PLAIN ENGLISH

Read the USE_SUPABASE environment variable. Default: "false" (local mode).

Convert to lowercase, check if it equals "true". Result is a boolean flag.

If you set USE_SUPABASE=true in your .env file...

...queries go to Supabase (cloud Postgres database). Perfect for Railway deployments.

Otherwise...

...queries go to local SQLite. No internet required, great for development.

Environment Variables: Your App's Secret Config

Your app reads its configuration from a .env file — a plain text file that stores sensitive keys and settings that should never be committed to Git. Here is what each one does:

MOONSHOT_API_KEY Your Moonshot AI key — required for query mode. Without it, every query returns a config error message.
COHERE_API_KEY Your Cohere reranking key — required for query mode. The reranker is not optional; removing it breaks retrieval.
USE_SUPABASE true = cloud Postgres (for production/Railway). false or absent = local SQLite (for development).
SUPABASE_URL + SUPABASE_KEY Only needed when USE_SUPABASE=true. Connects to your Supabase project.
⚠️
Never commit .env to Git

Your .gitignore already excludes .env. These API keys are essentially passwords — whoever has them can use your paid API quota. The .env.example file in your repo shows the required variable names without any real values, so collaborators know what to fill in.
